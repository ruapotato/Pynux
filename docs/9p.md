# 9P V0 ‚Äî wire spec for Hamnix

9P is the protocol Plan 9 uses for every file-server interaction. In
Hamnix, namespaces are per-process bindings to file servers (see
`docs/distro-namespaces.md`), and `mount(2)` takes an `srvfd` that
"speaks 9P" ‚Äî but until V0, nothing in the tree actually spoke it. This
document defines the wire format the codec library
(`lib/9p/9p.ad`) encodes and decodes. V1 wired that codec into
`sys/src/9/port/chan.ad::mount()`; later milestones made userspace
processes real 9P servers. This document is the contract those
follow-ons honour.

> **Status (2026-05):** 9P V0..V4.1 have all shipped ‚Äî see
> `STATUS.md`. The kernel client, per-process namespaces,
> `sys_srv_post`/`sys_srv_open`, and an end-to-end userspace-server ‚Üí
> kernel-client loop are all live.

The V0 dialect is **9P2000** ‚Äî the classic Plan 9 protocol described in
the Plan 9 4th edition `intro(5)` / `version(5)` / `attach(5)` / ‚Ä¶
pages. We deliberately do **not** ship 9P2000.L (the Linux-extension
dialect with `getattr`/`setattr`/`xattr`/`link`/`lock` etc.) or 9P2000.u
(the Unix-extensions dialect with numeric uid/gid and extension
strings). 9P2000 is the cleanest variant and is what every Plan 9 file
server speaks natively.

References we cross-checked while writing this:

  - Plan 9 4th edition manual, Volume 1 ¬ß5: `intro(5)`, `version(5)`,
    `attach(5)`, `walk(5)`, `open(5)`, `read(5)`, `stat(5)`,
    `flush(5)`, `error(5)`.
  - 9front sources `/sys/src/9/port/devmnt.c` (client) and
    `/sys/src/9/port/fcall.h` (message tags).
  - The libc9p header `<fcall.h>` and the
    `convM2S` / `convS2M` reference codec in `/sys/src/lib9p/`.
  - Russ Cox, *9P2000: protocol changes for the next-generation Plan
    9* (2002).

## 1. Framing and endianness

Every multi-byte integer field ‚Äî `size`, `tag`, `fid`, `length`, `qid`
components, file `mode`, every counted-length prefix ‚Äî is encoded
**little-endian**. There is no escape: 9P is unconditionally LE on the
wire regardless of host endianness. This matches every existing 9P
implementation we cross-checked.

Each message is a single contiguous byte string with a four-byte
self-describing length header:

```
size[4]   uint32   total bytes in this message INCLUDING the size field
type[1]   uint8    one of the message-type IDs in ¬ß3
tag[2]   uint16    request/reply correlator (see ¬ß2)
body...           type-specific payload
```

`size` ‚â• 7 always (the four `size` bytes plus `type` plus `tag`). The
maximum `size` a peer may emit is bounded by `msize`, the maximum
message size negotiated in `Tversion` (¬ß3.1). The Hamnix default
(per `lib/9p/9p.ad::P9_MSIZE_DEFAULT`) is **131072** bytes (128 KiB)
since the apt-install streaming refactor; the historical 8 KiB
default (which matches 9front and Linux `v9fs`) is the lower bound
a peer may negotiate down to. 128 KiB is sized so one full xz drain
chunk fits in a single Twrite.

Receivers MUST drop the connection if they read a `size` smaller than 7
or larger than the negotiated `msize`. The V0 codec's
`p9_decode_header` enforces the lower bound; the `msize` bound is the
caller's responsibility because the codec is mount-loop-agnostic.

> **Note:** the Hamnix codec's `P9_MSIZE_DEFAULT` was bumped from
> the historical 8 KiB to **128 KiB (131072 bytes)** in `4191250` so
> that `apt install libc6` (a 64 KiB tar stream per write) doesn't
> get split into ~8 Twrite round-trips per chunk. Both peers must
> agree; `9p_client.ad`'s `P9C_MSIZE`, `distrofs`'s buffer cap,
> `p9srv_demo`'s buffer cap, and `fs/pipe.ad` + `fs/socketpair.ad`'s
> ring sizes were all bumped together.

## 2. Tags and fids

### 2.1 Tags

Every T-message (request) carries a `tag` chosen by the client. The
matching R-message (reply) echoes that tag. The client uses the tag to
dispatch reply data to the right waiter ‚Äî there can be many concurrent
outstanding requests on one connection, so the protocol is intrinsically
multiplexed. Tags are scoped to one connection and may be reused once a
matching R-message arrives (or a `Tflush` clears it).

The reserved value `NOTAG = 0xFFFF` (`~0u16`) is used in exactly one
place: a `Tversion` message MUST use `tag = NOTAG`, and the matching
`Rversion` carries the same. This is a marker ‚Äî a peer rejecting a
`Tversion` doesn't know what tag domain the other side is using yet.

`Tflush.oldtag` (¬ß3.7) lets a client cancel a pending request; the
server replies with `Rflush` on the *new* tag, and a later `Rfoo` on
the cancelled `oldtag` MUST be discarded by the client.

### 2.2 Fids

A `fid` is a 32-bit handle the client uses to refer to a file (or auth
channel) on the server. **Fids are client-allocated**: when the client
walks a new file or auth-handshake, it picks a fresh fid value and
hands it to the server. The server then associates state (qid, open
mode, offset) with that fid for the life of the connection.

The reserved value `NOFID = 0xFFFFFFFF` (`~0u32`) means "no fid" and
appears in:

  - `Tattach.afid` when the client did not call `Tauth` first
    (i.e. the server allows attach-without-auth).
  - `Twalk.newfid` is **never** NOFID (a walk that lands on
    `nwname == 0` and `newfid == fid` clones in place; see ¬ß3.6).

Lifecycle of a fid:

  1. Born when `Tauth` / `Tattach` / `Twalk` succeeds with that fid as
     the destination. The client decides which integer to use; the
     server tracks it.
  2. Optionally opened via `Topen` or `Tcreate` ‚Äî this transitions the
     fid from "walking" to "I/O" state.
  3. Used for `Tread` / `Twrite` / `Tstat` / `Twstat`.
  4. Released by `Tclunk` (clean) or `Tremove` (clunk + unlink).
     Connection drop implicitly clunks every fid on that connection.

Reusing a fid that's still alive is a protocol error; the server
responds `Rerror "fid in use"`.

## 3. Message types

V0 ships every message-type pair the original 9P2000 specifies, even
when the kernel-side handler is deferred to V1. The codec lays bytes
down regardless; only the dispatch above it is feature-gated.

Numeric type IDs are the canonical 9P2000 values (matching 9front's
`/sys/src/9/port/fcall.h`):

| Code | Name      | Direction | Code | Name      |
|-----:|-----------|-----------|-----:|-----------|
|  100 | Tversion  | ‚Üí         |  101 | Rversion  |
|  102 | Tauth     | ‚Üí         |  103 | Rauth     |
|  104 | Tattach   | ‚Üí         |  105 | Rattach   |
|    ‚Äî | (no Terror)         |  107 | Rerror    |
|  108 | Tflush    | ‚Üí         |  109 | Rflush    |
|  110 | Twalk     | ‚Üí         |  111 | Rwalk     |
|  112 | Topen     | ‚Üí         |  113 | Ropen     |
|  114 | Tcreate   | ‚Üí         |  115 | Rcreate   |
|  116 | Tread     | ‚Üí         |  117 | Rread     |
|  118 | Twrite    | ‚Üí         |  119 | Rwrite    |
|  120 | Tclunk    | ‚Üí         |  121 | Rclunk    |
|  124 | Tstat     | ‚Üí         |  125 | Rstat     |
|  126 | Twstat    | ‚Üí         |  127 | Rwstat    |

Notice that 106 is **skipped** (the missing Terror ‚Äî see ¬ß3.4) and
122/123 (Tremove/Rremove) are NOT in this table because 9P2000 carries
`Tremove` at 122/123 in the canonical table, but V0 defers it under the
`stat`-write umbrella as Plan 9 does. The codec leaves space for
122/123 by name in V1; **V0 ships only the 12 pairs listed above**.

### 3.1 Tversion / Rversion (100/101) ‚Äî V0 SUPPORTED

```
Tversion: size[4] type[1]=100 tag[2]=NOTAG msize[4] version[s]
Rversion: size[4] type[1]=101 tag[2]=NOTAG msize[4] version[s]
```

First message on every fresh connection. The client picks a candidate
`msize` (‚â§ the buffer it can hold) and a candidate `version` string ‚Äî
for 9P2000 the literal ASCII `"9P2000"`. The server replies with an
`msize` ‚â§ what the client offered and a `version` that is either
`"9P2000"` (accepted) or `"unknown"` (rejected ‚Äî the client must close
the connection). Both messages use `tag = NOTAG` because no fid /
session state exists yet.

After a successful Rversion the connection is reset: all existing fids
on this connection are clunked, all pending tags are forgotten. A
client renegotiating mid-session uses this property to recover from a
desynchronised codec.

### 3.2 Tauth / Rauth (102/103) ‚Äî V0 SKELETON

```
Tauth:   size[4] type[1]=102 tag[2] afid[4] uname[s] aname[s]
Rauth:   size[4] type[1]=103 tag[2] aqid[13]
```

`Tauth` requests an auth channel ‚Äî a fid the client will read/write
through to perform whatever challenge/response the server's auth
scheme demands. `afid` is the client-allocated fid for that channel.
`uname` is the user the client wants to authenticate as; `aname` is the
file-tree the client intends to `Tattach` once auth completes (servers
may serve multiple trees on one connection ‚Äî `aname` selects one).

Rauth carries a `qid` describing the auth channel (the file the client
now reads/writes through `afid`). If the server requires no auth it
replies `Rerror "authentication not required"` and the client uses
`NOFID` in the subsequent `Tattach.afid`.

**V0 ships the encode/decode bodies but the kernel-side
handler stays unimplemented.** A later milestone lifts this when a
9P server learns to authenticate; until then `Tauth` is best-encoded
by clients that want to fault-test the codec, not by a working
namespace mount.

### 3.3 Tattach / Rattach (104/105) ‚Äî V0 SUPPORTED

```
Tattach: size[4] type[1]=104 tag[2] fid[4] afid[4] uname[s] aname[s]
Rattach: size[4] type[1]=105 tag[2] qid[13]
```

Binds `fid` to the root of the server's `aname` file tree, on behalf
of user `uname`. `afid` is either the post-`Rauth` auth fid or
`NOFID`. The reply's `qid` is the root's qid.

After a successful `Rattach`, every other operation walks from `fid`
to the file the client wants to act on.

### 3.4 Rerror (107) ‚Äî V0 SUPPORTED

```
Rerror:  size[4] type[1]=107 tag[2] ename[s]
```

**There is no `Terror`.** 9P uses Rerror as the universal failure
reply: any T-message can be answered with an Rerror that carries the
same tag. The `ename` is a UTF-8 error string (no NUL), matching the
per-process error-string convention Hamnix already uses
(`sys/src/9/port/error.ad`'s `set_current_errstr` / `errstr(2)`).
Servers SHOULD keep `ename` ‚â§ 64 bytes for parity with what
`errstr(2)` returns; nothing in the protocol enforces that bound.

### 3.5 Tflush / Rflush (108/109) ‚Äî V0 SUPPORTED

```
Tflush:  size[4] type[1]=108 tag[2] oldtag[2]
Rflush:  size[4] type[1]=109 tag[2]
```

Cancels a request with tag `oldtag`. The server eventually replies
`Rflush`; from that point onwards the client MUST discard any
`Rfoo` arriving on `oldtag`. The cancelled request may already have
completed ‚Äî `Rflush` does not promise the request was rolled back, only
that any in-flight reply for it is invalidated.

### 3.6 Twalk / Rwalk (110/111) ‚Äî V0 SUPPORTED

```
Twalk:   size[4] type[1]=110 tag[2] fid[4] newfid[4]
         nwname[2] (wname[s])*nwname
Rwalk:   size[4] type[1]=111 tag[2] nwqid[2] (wqid[13])*nwqid
```

Walks from `fid` through up to **16** path components, depositing the
resulting handle at `newfid`. If `newfid == fid` the walk clones in
place (no second fid is allocated). Partial walks succeed: the reply
returns `nwqid` qids ‚Äî one per name component that the server could
follow. If `nwqid == nwname`, the walk fully succeeded; otherwise the
client clunks `newfid` (the partial state) and either retries or gives
up.

**Twalk MUST list up to 16 names** ‚Äî both peers MUST reject more. This
bound is what keeps the on-wire message bounded by `msize` for
reasonable name lengths.

A walk with `nwname == 0` is the canonical "clone fid" operation:
duplicate the file `fid` references into `newfid` so the client can
walk further from a fresh handle without losing the original.

### 3.7 Topen / Ropen (112/113) ‚Äî V0 SUPPORTED

```
Topen:   size[4] type[1]=112 tag[2] fid[4] mode[1]
Ropen:   size[4] type[1]=113 tag[2] qid[13] iounit[4]
```

Transitions `fid` from "walked but not opened" to "open for I/O". The
`mode` byte is the Plan 9 open mode (`OREAD/OWRITE/ORDWR/OEXEC` in the
low two bits, plus `OTRUNC=0x10`, `OCEXEC=0x20`, `ORCLOSE=0x40`). The
`iounit` field tells the client the maximum byte count it should pass
to `Tread.count` / `Twrite.count` for this fid ‚Äî 0 means "use msize
minus headers". Hamnix's V1 server-side handler should compute
`iounit = msize - 24` for regular files.

### 3.8 Tcreate / Rcreate (114/115) ‚Äî V0 SUPPORTED

```
Tcreate: size[4] type[1]=114 tag[2] fid[4] name[s] perm[4] mode[1]
Rcreate: size[4] type[1]=115 tag[2] qid[13] iounit[4]
```

Creates `name` inside the directory `fid` references, then opens it as
if by `Topen`. `perm` carries Plan 9 mode bits: low 9 are rwxrwxrwx, plus
`DMDIR = 0x80000000` (create as directory), `DMAPPEND = 0x40000000`,
`DMEXCL = 0x20000000`. `mode` is the same byte as in ¬ß3.7. `fid` is
mutated by this call ‚Äî after a successful Rcreate, `fid` now references
the newly-created file (NOT the parent directory).

### 3.9 Tread / Rread (116/117) ‚Äî V0 SUPPORTED

```
Tread:   size[4] type[1]=116 tag[2] fid[4] offset[8] count[4]
Rread:   size[4] type[1]=117 tag[2] count[4] data[count]
```

Reads up to `count` bytes from `fid` starting at byte `offset`. The
reply's `count` is how many bytes were actually delivered; `data` is
the raw bytes. For directory fids, `data` is a sequence of `stat`
records (¬ß4) and `offset` MUST be either 0 (rewind) or the byte
position of a previously-returned record boundary ‚Äî never a value
mid-record.

### 3.10 Twrite / Rwrite (118/119) ‚Äî V0 SUPPORTED

```
Twrite:  size[4] type[1]=118 tag[2] fid[4] offset[8] count[4] data[count]
Rwrite:  size[4] type[1]=119 tag[2] count[4]
```

Writes `count` bytes at `offset`. The reply's `count` is what the
server accepted (may be less than the request's). On a fid opened with
`OAPPEND`, the server ignores `offset` and always appends.

### 3.11 Tclunk / Rclunk (120/121) ‚Äî V0 SUPPORTED

```
Tclunk:  size[4] type[1]=120 tag[2] fid[4]
Rclunk:  size[4] type[1]=121 tag[2]
```

Releases the server-side state for `fid`. After Rclunk, `fid` may be
reused for a fresh walk/auth/attach. A clunk on an open fid does NOT
flush pending I/O ‚Äî the server may discard buffered writes ‚Äî so
well-behaved clients fsync (or its equivalent ‚Äî there's no Tfsync;
the closest is `Twstat` writing the length) before clunking.

### 3.12 Tstat / Rstat (124/125) ‚Äî V0 SUPPORTED

```
Tstat:   size[4] type[1]=124 tag[2] fid[4]
Rstat:   size[4] type[1]=125 tag[2] stat[n]
```

Returns the `stat` record (¬ß4) describing the file `fid` references.
The `stat[n]` payload is a 2-byte length prefix followed by that many
bytes of the ¬ß4 Stat layout ‚Äî note the outer Rstat already has its own
`size[4]` header, so the inner length prefix is genuinely redundant
*except* that some 9P implementations (notably Plan 9's `lib9p`)
re-use the wire-form Stat directly without unwrapping; we honour that
convention.

### 3.13 Twstat / Rwstat (126/127) ‚Äî V0 SKELETON

```
Twstat:  size[4] type[1]=126 tag[2] fid[4] stat[n]
Rwstat:  size[4] type[1]=127 tag[2]
```

Writes a Stat record back to the server, requesting the server update
the file's metadata. Fields whose stored value should be unchanged
are encoded with a **sentinel value** (`~0` for numeric types, empty
string for `s` types). The server MUST reject changes that would
violate permissions; the canonical errors are described in `stat(5)`.

**V0 ships the encode/decode bodies but no kernel-side handler.** V1
will wire `Twstat` to `wstat(2)` (`sys/src/9/port/sysfile.ad`'s
`do_wstat`).

## 4. Qid and Stat layout

### 4.1 Qid (13 bytes on the wire)

```
qid.type[1]    file-class bits (see below)
qid.vers[4]    version number (bumped on every write)
qid.path[8]    server-unique inode-style identifier
```

The qid uniquely names a file on the server *for the life of the
server*. `qid.path` is unique across all files; `qid.vers` is bumped
every time the file is written so a stale handle can detect change.

`qid.type` bit assignments (9P2000 canonical):

```
0x80   QTDIR     directory
0x40   QTAPPEND  append-only file
0x20   QTEXCL    exclusive-use file
0x10   QTMOUNT   mounted channel (server-side bookkeeping)
0x08   QTAUTH    authentication channel
0x04   QTTMP     temporary file (no backup)
0x00   QTFILE    regular file (no bits set)
```

**V0 supports `QTDIR` (0x80) and `QTFILE` (0x00) only.** The other bits
are accepted on input (the decoder doesn't validate them away) but the
spec defers honouring them to V1.

### 4.2 Stat record

The Stat record is what `Rstat.stat[n]` carries and what `Tread` on a
directory streams. Its layout ‚Äî including the **outer 2-byte length
prefix** ‚Äî is:

```
size[2]                                outer length (record bytes minus 2)
type[2]                                server-defined kind (Plan 9: 0)
dev[4]                                 server-defined dev (Plan 9: 0)
qid.type[1] qid.vers[4] qid.path[8]    13 bytes
mode[4]                                Plan 9 perm bits + DM* high bits
atime[4]                               seconds since epoch
mtime[4]                               seconds since epoch
length[8]                              file size in bytes
name[s]                                file basename, UTF-8 no NUL
uid[s]                                 owner user name
gid[s]                                 owning group name
muid[s]                                user who last modified
```

This is **exactly** the byte layout `docs/native-api.md` references
under "Directory format", and exactly what
`sys/src/9/port/sysfile.ad::_write_dir_record` emits ‚Äî the V0 codec
reuses that layout verbatim so kernel-side stat readers and on-wire
9P stat readers parse the same bytes.

The four counted strings at the tail are each encoded as `length[2]`
(little-endian uint16, the byte count, not the codepoint count)
followed by `length` UTF-8 bytes with no terminating NUL.

For V0, all four strings on Hamnix-served files are the literal ASCII
`"hamnix"` ‚Äî there's no user-account database yet. V1 will switch
`uid`/`gid`/`muid` to per-process identity once `/etc/passwd` exists.

### 4.3 Wstat semantics (encoding only in V0)

A Wstat record uses the same layout as a Stat record, but every field
the client does NOT want to change MUST carry its sentinel:

  - Numeric fields (`type`, `dev`, `mode`, `atime`, `mtime`, `length`,
    each `qid.*`): the all-ones value at that field's width
    (`0xFFFF`, `0xFFFFFFFF`, `0xFFFFFFFFFFFFFFFF`). The codec exposes
    `P9_WSTAT_*_UNCHANGED` constants for callers.
  - String fields (`name`, `uid`, `gid`, `muid`): empty string ‚Äî
    length prefix 0, no bytes.

V0 ships the encoder/decoder; the kernel does not interpret these
sentinels yet (that's V1).

## 5. Counted lengths (`s` and `b`)

9P uses two count-prefixed types in its body grammar:

  - **`s`** ‚Äî a UTF-8 string. Encoded as `length[2]` (little-endian
    uint16, the byte count) followed by `length` bytes. **No NUL
    terminator.** The maximum representable length is 65535 ‚Äî V0 caps
    at this. Path components inside `Twalk.wname` are limited to 255
    bytes by convention but the wire format allows up to 65535.

  - **`b`** ‚Äî a counted byte array, used only inside `Tread.data` /
    `Twrite.data` / `Rread.data`. Encoded as `length[4]`
    (little-endian uint32, the byte count) followed by `length` bytes.

The codec exposes `p9_emit_string` / `p9_decode_string` for `s` and
treats the `b` payload as raw bytes, since every caller cares about
the surrounding fid/offset context.

## 6. V0 scope summary

The V0 codec at `lib/9p/9p.ad` ships full encode + decode for every
message type listed in ¬ß3. Twelve of those (the **V0 SUPPORTED**
items) will be wired into the kernel's mount loop in V1; two
(**V0 SKELETON**: `Tauth`/`Rauth`, `Twstat`/`Rwstat`) ship as codec
bytes only.

V1 deliverables (separately tracked):

  - `sys/src/9/port/chan.ad::mount()` issues `Tversion` then
    `Tattach` over `srvfd` using the V0 codec.
  - `sys/src/9/port/syschan.ad` walks/opens/reads/writes/clunks
    against mounted channels.
  - A userspace daemon (e.g. `rio`) becomes a real 9P server.
  - Tauth + Twstat get real kernel-side bodies.

V0 ships the protocol definition and the byte-level primitives; the
kernel-side adoption is the work the next milestones do on top.
