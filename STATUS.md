# Hamnix STATUS — milestone log

The full milestone trajectory across three tracks: the bare-metal Hamnix
kernel (M-series), the Linux ABI compatibility track (L-series), and the
Linux userspace ABI track (U-series).

**Status legend:**
- **Done** — landed on `main` with a passing test.
- **Partial** — landed but a documented gap remains.
- **Diagnosis** — investigated, documented blocker, no code change.
- **Pending** — not yet started.

For the current end-state plus open blockers, see [README.md](README.md).

| M16.97 | ICMP echo round-trip — finishes the M16.96 scaffold. Root cause was two Adder codegen quirks in `arp_lookup`: (1) `cast[uint64](arp_ip[i])` on an `Array[N, uint32]` global doesn't zero-extend cleanly across `==`, so the equality compare reported "not equal" even when bit-patterns matched; (2) `&arp_mac[i][0]` on the 2-D `Array[N, Array[6, uint8]]` MAC table lowered to address 0 instead of the correct slot, so `ip_send` saw `dst_mac == NULL` and dropped every outbound IP frame. | **preemptive scheduling** | Hamnix multitasked **purely cooperatively** — a task yielded the CPU only inside a blocking syscall (`waitpid`, a blocking pipe/stdin read, `yield`) — so a tight-looping userland task that made no blocking syscall starved every other task forever; one bad process hung the box. `timer_interrupt()` already called `schedule()` on every 100 Hz tick, but it switched out of *any* context (including ring-0 kthread / IRQ code, where preempting half-updated kernel state is unsafe), and the real workload never had two concurrent `STATE_READY` user tasks, so the switch always early-returned.
