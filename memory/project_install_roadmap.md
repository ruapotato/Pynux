Hamnix is on a path toward a real installation on physical hardware,
not just QEMU. Two filesystem drivers + UEFI are the long-term work
that unlocks that.

**Why:** Real-hardware boot via UEFI requires reading the kernel from
an EFI System Partition (FAT32) and mounting a root filesystem
(EXT4 is the realistic Linux-compatible target). Without these,
Hamnix can demo on QEMU forever but never boots a laptop.

**How to apply:** Queue these as milestones AFTER the current
shell/userland buildout. Status as of 2026-05-15:

  1. Read-only FAT32 driver — SHIPPED M16.43..M16.46. Mounts at
     /mnt via either baked ramdisk (always) or virtio-blk-attached
     image (`-drive disk.img,if=virtio`). Subdirectory traversal,
     listdir, multi-component path lookup all work. Image is
     hand-built in Python by scripts/build_diskimg.py (no mtools
     dependency).
  2. UEFI boot loader / chainloader path — NOT STARTED. Real
     hardware boot still depends on this; QEMU stays on
     multiboot1 (`qemu -kernel`). Will need a PE-format .efi
     binary, EFI Boot Services calls, ExitBootServices, then
     hand-off to the existing kernel entry point.
  3. EXT4 read-only — SHIPPED M16.51..M16.58. mkfs.ext4 image
     (generated by build_diskimg.py via debugfs, no sudo / loop
     mount), attached via virtio-blk, mounts at /ext. Handles
     superblock, group descriptors (32 AND 64-byte), inode read
     via inodes_per_group, INDEX extents (eh_depth > 0 walked
     transparently via a 5-level-bounded loop, M16.58), 48-bit
     physical addresses (ee_start_hi / ei_leaf_hi honored),
     ext4_dir_entry_2 dirent walk, multi-component path lookup,
     /ls /ext routing. NOT yet: multi-block directories (current
     dir code reads only the first data block — ~30 entries in
     1 KiB), htree directory indexes.
  4. EXT4 write — NOT STARTED. Last step before real-laptop boot.
     Natural decomposition: block bitmap allocator → small file
     create in root dir → file extension via new extent → dir
     entry add/remove. has_journal already disabled at mkfs time
     so the journal write path is not blocking.

Surprise: the VFS DIDN'T need a superblock/inode/dentry refactor
to add EXT4. We extended the existing prefix-routing approach
(is_fat_path → fat_*, is_ext_path → ext4_*) with FD_FAT_MARK /
FD_EXT4_MARK markers, mirroring how /proc and /tmp already
worked. The refactor becomes valuable when there are >2 mounts of
the same FS or when readdir needs a unified backend, neither of
which we've hit yet.
